# 任務: 怪手圖示測驗應用程式

**來源**: 設計文件位於 `/specs/001-create-quiz-app/`
**先決條件**: plan.md (必要), spec.md (必要)

**測試**: 由於規格中未明確要求，暫不包含自動化測試任務。重點將放在功能的實現上。

**組織**: 任務按使用者故事分組，以便獨立實現和測試每個故事。

## 格式: `[ID] [P?] [Story] 描述`

- **[P]**: 可並行執行的任務
- **[Story]**: 該任務所屬的使用者故事 (例如, US1, US2)

---

## 階段一: 環境設定 (共享基礎設施)

**目的**: 初始化專案並建立基礎結構。

- [x] T001 [P] 在專案根目錄下，使用 `npm create vite@latest . -- --template react` 初始化 Vite 專案。
- [x] T002 [P] 使用 `npm install` 安裝 React 相關依賴。
- [x] T003 [P] 使用 `npm install -D tailwindcss postcss autoprefixer` 安裝 Tailwind CSS 開發依賴。
- [x] T004 [P] 執行 `npx tailwindcss init -p` 來產生 `tailwind.config.js` 和 `postcss.config.js` 檔案。
- [x] T005 更新 `tailwind.config.js` 的 `content` 欄位以包含 `src` 目錄下的所有 `jsx` 檔案。
- [x] T006 在 `src/index.css` 中加入 Tailwind CSS 的 `@tailwind` 指令。

---

## 階段二: 基礎建設 (阻塞性先決條件)

**目的**: 建立所有使用者故事都依賴的核心應用程式架構。
**⚠️ 關鍵**: 在此階段完成前，任何使用者故事的開發都不能開始。

- [x] T007 建立主應用程式元件 `src/App.jsx`，並設定基礎的狀態管理邏輯（例如，`gameState`，`score`）。
- [x] T008 [P] 建立 `src/components` 資料夾，並在其中建立空的元件檔案：`StartScreen.jsx`, `Quiz.jsx`, `Results.jsx`。
- [x] T009 在 `src/App.jsx` 中，使用 `useEffect` hook 實現非同步邏輯，以在應用程式啟動時從 `public/icon_mapping.json` 載入測驗資料。
- [x] T010 在 `src/App.jsx` 中，根據 `gameState`（'start', 'quiz', 'results'）的狀態，實現條件渲染邏輯，以顯示對應的元件。

**檢查點**: 基礎建設完成。使用者故事的開發現在可以並行開始。

---

## 階段三: 使用者故事 1 - 開始測驗 (優先度: P1) 🎯 MVP

**目標**: 讓使用者可以從一個歡迎畫面開始測驗。
**獨立測試**: 應用程式載入後顯示開始畫面，點擊「開始」按鈕後，畫面轉換至第一個測驗問題。

### 實作任務

- [x] T011 [US1] 實作 `src/components/StartScreen.jsx` 的 UI，包含標題和一個「開始測驗」按鈕。
- [x] T012 [US1] 從 `App.jsx` 向 `StartScreen.jsx` 傳遞一個 `onStart` 函數，用於在按鈕被點擊時將 `gameState` 從 'start' 更新為 'quiz'。

**檢查點**: 使用者故事 1 此時應可獨立運作並進行測試。

---

## 階段四: 使用者故事 2 - 回答問題 (優先度: P2)

**目標**: 逐一呈現問題並處理使用者的回答。
**獨立測試**: 在問題畫面中，選擇一個答案後，測驗會前進到下一個問題或結果畫面。

### 實作任務

- [x] T013 [US2] 實作 `src/components/Quiz.jsx` 的 UI，用於顯示問題圖片和多個選項按鈕。
- [x] T014 [US2] 從 `App.jsx` 向 `Quiz.jsx` 傳遞當前問題的資料物件。
- [x] T015 [US2] 在 `Quiz.jsx` 中實現邏輯，以顯示問題圖片和包含一個正確答案及數個隨機干擾選項的按鈕。
- [x] T016 [US2] 從 `App.jsx` 向 `Quiz.jsx` 傳遞一個 `onAnswer` 處理函數。
- [x] T017 [US2] 在 `App.jsx` 中實現 `onAnswer` 函數，用於檢查答案、更新分數、並將索引推進到下一個問題；如果是最後一題，則將 `gameState` 更新為 'results'。

**檢查點**: 使用者故事 1 和 2 此時應可協同運作。

---

## 階段五: 使用者故事 3 - 查看結果與學習 (優先度: P3)

**目標**: 顯示最終分數和一份完整的答案解析。
**獨立測試**: 回答完最後一題後，結果畫面會顯示正確的分數和所有問題的答案列表。

### 實作任務

- [x] T018 [US3] 實作 `src/components/Results.jsx` 的 UI。
- [x] T019 [US3] 從 `App.jsx` 向 `Results.jsx` 傳遞最終分數和完整的題庫列表。
- [x] T020 [US3] 在 `Results.jsx` 中顯示最終分數，並遍歷題庫列表以渲染出包含答案圖片和正確名稱的答案解析。
- [x] T021 [US3] 在 `Results.jsx` 中加入一個「重新開始」按鈕，並從 `App.jsx` 傳遞一個 `onRestart` 函數以重設所有狀態並將 `gameState` 切換回 'start'。

**檢查點**: 所有使用者故事此時應可獨立運作。

---

## 階段六: 優化與跨領域關注點

**目的**: 改善整體應用程式的品質。

- [x] T022 [P] 審查並優化所有元件的 Tailwind CSS，確保在桌面和行動裝置上都有良好的響應式表現。
- [x] T023 [P] （可選）為元件切換或UI互動加入細微的轉場或動畫效果，提升使用者體驗。
- [x] T024 進行最終的程式碼審查和清理。

---

## 依賴性與執行順序

- **階段一 (環境設定)**: 無依賴，可立即開始。
- **階段二 (基礎建設)**: 依賴階段一完成。**阻塞**所有使用者故事的開發。
- **使用者故事 (階段三、四、五)**: 均依賴階段二完成。完成後，理論上可以並行開發，但建議按 P1 -> P2 -> P3 的順序進行以確保核心流程穩定。

---

## 實施策略

### MVP 優先 (僅使用者故事 1)

1. 完成階段一：環境設定
2. 完成階段二：基礎建設
3. 完成階段三：使用者故事 1
4. **停止並驗證**: 獨立測試使用者故事 1 的功能。

### 增量交付

1. 完成階段一 + 階段二 → 基礎就緒。
2. 加入使用者故事 1 → 獨立測試 → 可交付 MVP。
3. 加入使用者故事 2 → 獨立測試 → 可交付更新。
4. 加入使用者故事 3 → 獨立測試 → 交付最終產品。
5. 每個故事都為產品增加價值，且不破壞先前功能。
